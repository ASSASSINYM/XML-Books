<p><strong>Extensible Markup Language</strong>&nbsp;(<strong>XML</strong>) is a&nbsp;<a title="Markup language" href="https://en.wikipedia.org/wiki/Markup_language">markup language</a>&nbsp;that defines a set of rules for encoding&nbsp;<a title="Electronic document" href="https://en.wikipedia.org/wiki/Electronic_document">documents</a>&nbsp;in a&nbsp;<a title="File format" href="https://en.wikipedia.org/wiki/File_format">format</a>&nbsp;that is both&nbsp;<a title="Human-readable medium" href="https://en.wikipedia.org/wiki/Human-readable_medium">human-readable</a>&nbsp;and&nbsp;<a title="Machine-readable data" href="https://en.wikipedia.org/wiki/Machine-readable_data">machine-readable</a>. The&nbsp;<a title="World Wide Web Consortium" href="https://en.wikipedia.org/wiki/World_Wide_Web_Consortium">World Wide Web Consortium</a>'s XML 1.0 Specification&nbsp;of 1998&nbsp;and several other related specifications &mdash;all of them free&nbsp;<a title="Open standard" href="https://en.wikipedia.org/wiki/Open_standard">open standards</a>&mdash;define XML.</p>
<p>The design goals of XML emphasize simplicity, generality, and usability across the&nbsp;<a title="Internet" href="https://en.wikipedia.org/wiki/Internet">Internet</a>.&nbsp;It is a textual data format with strong support via&nbsp;<a title="Unicode" href="https://en.wikipedia.org/wiki/Unicode">Unicode</a>&nbsp;for different&nbsp;<a title="Language" href="https://en.wikipedia.org/wiki/Language">human languages</a>. Although the design of XML focuses on documents, the language is widely used for the representation of arbitrary&nbsp;<a title="Data structure" href="https://en.wikipedia.org/wiki/Data_structure">data structures</a>&nbsp;such as those used in&nbsp;<a title="Web service" href="https://en.wikipedia.org/wiki/Web_service">web services</a>.</p>
<p>Several&nbsp;<a title="XML schema" href="https://en.wikipedia.org/wiki/XML_schema">schema systems</a>&nbsp;exist to aid in the definition of XML-based languages, while programmers have developed many&nbsp;<a title="Application programming interface" href="https://en.wikipedia.org/wiki/Application_programming_interface">application programming interfaces</a>&nbsp;(APIs) to aid the processing of XML data.</p>

<h2><span id="Applications" class="mw-headline">Applications</span></h2>
<p>The essence of why extensible markup languages are necessary is explained at&nbsp;<em><a title="Markup language" href="https://en.wikipedia.org/wiki/Markup_language">Markup language</a></em>&nbsp;(for example, see&nbsp;<em><a title="Markup language" href="https://en.wikipedia.org/wiki/Markup_language#XML">Markup language &sect; XML</a></em>) and at&nbsp;<em><a title="Standard Generalized Markup Language" href="https://en.wikipedia.org/wiki/Standard_Generalized_Markup_Language">Standard Generalized Markup Language</a></em>.</p>
<p>Hundreds of document formats using XML syntax have been developed,&nbsp;including&nbsp;<a title="RSS" href="https://en.wikipedia.org/wiki/RSS">RSS</a>,&nbsp;<a class="mw-redirect" title="Atom (standard)" href="https://en.wikipedia.org/wiki/Atom_(standard)">Atom</a>,&nbsp;<a title="SOAP" href="https://en.wikipedia.org/wiki/SOAP">SOAP</a>,&nbsp;<a class="mw-redirect" title="SVG" href="https://en.wikipedia.org/wiki/SVG">SVG</a>, and&nbsp;<a title="XHTML" href="https://en.wikipedia.org/wiki/XHTML">XHTML</a>. XML-based formats have become the default for many office-productivity tools, including&nbsp;<a title="Microsoft Office" href="https://en.wikipedia.org/wiki/Microsoft_Office">Microsoft Office</a>&nbsp;(<a title="Office Open XML" href="https://en.wikipedia.org/wiki/Office_Open_XML">Office Open XML</a>),&nbsp;<a title="OpenOffice.org" href="https://en.wikipedia.org/wiki/OpenOffice.org">OpenOffice.org</a>&nbsp;and&nbsp;<a title="LibreOffice" href="https://en.wikipedia.org/wiki/LibreOffice">LibreOffice</a>&nbsp;(<a title="OpenDocument" href="https://en.wikipedia.org/wiki/OpenDocument">OpenDocument</a>), and&nbsp;<a class="mw-redirect" title="Apple Computer" href="https://en.wikipedia.org/wiki/Apple_Computer">Apple</a>'s&nbsp;<a title="IWork" href="https://en.wikipedia.org/wiki/IWork">iWork</a>. XML has also provided the base language for&nbsp;<a title="Communication protocol" href="https://en.wikipedia.org/wiki/Communication_protocol">communication protocols</a>&nbsp;such as&nbsp;<a class="mw-redirect" title="Extensible Messaging and Presence Protocol" href="https://en.wikipedia.org/wiki/Extensible_Messaging_and_Presence_Protocol">XMPP</a>. Applications for the&nbsp;<a title="Microsoft" href="https://en.wikipedia.org/wiki/Microsoft">Microsoft</a>&nbsp;<a title=".NET Framework" href="https://en.wikipedia.org/wiki/.NET_Framework">.NET Framework</a>&nbsp;use XML files for configuration, and&nbsp;<a title="Property list" href="https://en.wikipedia.org/wiki/Property_list">property lists</a>&nbsp;are an implementation of configuration storage built on XML.</p>
<p>Many industry data standards, such as&nbsp;<a title="Health Level 7" href="https://en.wikipedia.org/wiki/Health_Level_7">Health Level 7</a>,&nbsp;<a title="OpenTravel Alliance" href="https://en.wikipedia.org/wiki/OpenTravel_Alliance">OpenTravel Alliance</a>,&nbsp;<a title="FpML" href="https://en.wikipedia.org/wiki/FpML">FpML</a>,&nbsp;<a title="MISMO" href="https://en.wikipedia.org/wiki/MISMO">MISMO</a>, and&nbsp;<a title="National Information Exchange Model" href="https://en.wikipedia.org/wiki/National_Information_Exchange_Model">National Information Exchange Model</a>&nbsp;are based on XML and the rich features of the XML schema specification. Many of these standards are quite complex and it is not uncommon for a specification to comprise several thousand pages.&nbsp;In publishing,&nbsp;<a title="Darwin Information Typing Architecture" href="https://en.wikipedia.org/wiki/Darwin_Information_Typing_Architecture">Darwin Information Typing Architecture</a>&nbsp;is an XML industry data standard. XML is used extensively to underpin various publishing formats.</p>
<p>XML is widely used in a&nbsp;<a title="Service-oriented architecture" href="https://en.wikipedia.org/wiki/Service-oriented_architecture">Service-oriented architecture</a>&nbsp;(SOA). Disparate systems communicate with each other by exchanging XML messages. The message exchange format is standardised as an&nbsp;<a title="XML schema" href="https://en.wikipedia.org/wiki/XML_schema">XML schema</a>&nbsp;(XSD). This is also referred to as the canonical schema. XML has come into common use for the interchange of data over the Internet.&nbsp;<a title="History of the Internet" href="https://en.wikipedia.org/wiki/History_of_the_Internet#Internet_Engineering_Task_Force">IETF</a>&nbsp;<a class="extiw" title="rfc:3023" href="https://tools.ietf.org/html/rfc3023">RFC:3023</a>, now superseded by&nbsp;<a class="extiw" title="rfc:7303" href="https://tools.ietf.org/html/rfc7303">RFC:7303</a>, gave rules for the construction of&nbsp;<a class="mw-redirect" title="Internet media type" href="https://en.wikipedia.org/wiki/Internet_media_type">Internet Media Types</a>&nbsp;for use when sending XML. It also defines the media types&nbsp;<code>application/xml</code>&nbsp;and&nbsp;<code>text/xml</code>, which say only that the data is in XML, and nothing about its&nbsp;<a title="Semantics" href="https://en.wikipedia.org/wiki/Semantics">semantics</a>.</p>
<p><a class="external mw-magiclink-rfc" href="https://tools.ietf.org/html/rfc7303" rel="nofollow">RFC 7303</a>&nbsp;also recommends that XML-based languages be given media types ending in&nbsp;<code>+xml</code>; for example&nbsp;<code>image/svg+xml</code>&nbsp;for&nbsp;<a class="mw-redirect" title="SVG" href="https://en.wikipedia.org/wiki/SVG">SVG</a>. Further guidelines for the use of XML in a networked context appear in&nbsp;<a class="external mw-magiclink-rfc" href="https://tools.ietf.org/html/rfc3470" rel="nofollow">RFC 3470</a>, also known as IETF BCP 70, a document covering many aspects of designing and deploying an XML-based language.</p>
<h2><span id="Key_terminology" class="mw-headline">Key terminology</span></h2>
<p>The material in this section is based on the XML Specification. This is not an exhaustive list of all the constructs that appear in XML; it provides an introduction to the key constructs most often encountered in day-to-day use.</p>
<dl>
<dt>Character</dt>
<dd>An XML document is a string of&nbsp;<em>characters</em>.&nbsp;<a title="Valid characters in XML" href="https://en.wikipedia.org/wiki/Valid_characters_in_XML">Almost</a>&nbsp;every legal&nbsp;<a title="Unicode" href="https://en.wikipedia.org/wiki/Unicode">Unicode</a>&nbsp;character may appear in an XML document.</dd>
</dl>
<dl>
<dt>Processor and application</dt>
<dd>The&nbsp;<em>processor</em>&nbsp;analyzes the markup and passes structured information to an&nbsp;<em>application</em>. The specification places requirements on what an XML processor must do and not do, but the application is outside its scope. The processor (as the specification calls it) is often referred to colloquially as an&nbsp;<em>XML parser</em>.</dd>
</dl>
<dl>
<dt>Markup and content</dt>
<dd>The characters making up an XML document are divided into&nbsp;<em>markup</em>&nbsp;and&nbsp;<em>content</em>, which may be distinguished by the application of simple syntactic rules. Generally, strings that constitute markup either begin with the character&nbsp;<code>&lt;</code>&nbsp;and end with a&nbsp;<code>&gt;</code>, or they begin with the character&nbsp;<code>&amp;</code>&nbsp;and end with a&nbsp;<code>;</code>. Strings of characters that are not markup are content. However, in a&nbsp;<a title="CDATA" href="https://en.wikipedia.org/wiki/CDATA">CDATA</a>&nbsp;section, the delimiters&nbsp;<code>&lt;![CDATA[</code>&nbsp;and&nbsp;<code>]]&gt;</code>&nbsp;are classified as markup, while the text between them is classified as content. In addition, whitespace before and after the outermost element is classified as markup.</dd>
</dl>
<dl>
<dt>Tag</dt>
<dd>A&nbsp;<em>tag</em>&nbsp;is a markup construct that begins with&nbsp;<code>&lt;</code>&nbsp;and ends with&nbsp;<code>&gt;</code>. Tags come in three flavors:
<ul>
<li><em>start-tag</em>, such as&nbsp;<code>&lt;section&gt;</code>;</li>
<li><em>end-tag</em>, such as&nbsp;<code>&lt;/section&gt;</code>;</li>
<li><em>empty-element tag</em>, such as&nbsp;<code>&lt;line-break /&gt;</code>.</li>
</ul>
</dd>
</dl>
<dl>
<dt>Element</dt>
<dd>An&nbsp;<em>element</em>&nbsp;is a logical document component that either begins with a start-tag and ends with a matching end-tag or consists only of an empty-element tag. The characters between the start-tag and end-tag, if any, are the element's&nbsp;<em>content</em>, and may contain markup, including other elements, which are called&nbsp;<em>child elements</em>. An example is&nbsp;<code>&lt;greeting&gt;Hello, world!&lt;/greeting&gt;</code>. Another is&nbsp;<code>&lt;line-break&nbsp;/&gt;</code>.</dd>
</dl>
<dl>
<dt>Attribute</dt>
<dd>An&nbsp;<em>attribute</em>&nbsp;is a markup construct consisting of a name&ndash;value pair that exists within a start-tag or empty-element tag. An example is&nbsp;<code>&lt;img src="madonna.jpg" alt="Madonna" /&gt;</code>, where the names of the attributes are "src" and "alt", and their values are "madonna.jpg" and "Madonna" respectively. Another example is&nbsp;<code>&lt;step number="3"&gt;Connect A to B.&lt;/step&gt;</code>, where the name of the attribute is "number" and its value is "3". An XML attribute can only have a single value and each attribute can appear at most once on each element. In the common situation where a list of multiple values is desired, this must be done by encoding the list into a well-formed XML attribute&nbsp;with some format beyond what XML defines itself. Usually this is either a comma or semi-colon delimited list or, if the individual values are known not to contain spaces, a space-delimited list can be used.&nbsp;<code>&lt;div class="inner greeting-box"&gt;Welcome!&lt;/div&gt;</code>, where the attribute "class" has both the value "inner greeting-box" and also indicates the two&nbsp;<a class="mw-redirect" title="CSS" href="https://en.wikipedia.org/wiki/CSS">CSS</a>&nbsp;class names "inner" and "greeting-box".</dd>
</dl>
<dl>
<dt>XML declaration</dt>
<dd>XML documents may begin with an&nbsp;<em>XML declaration</em>&nbsp;that describes some information about themselves. An example is&nbsp;<code>&lt;?xml version="1.0" encoding="UTF-8"?&gt;</code>.</dd>
</dl>
<h2><span id="Characters_and_escaping" class="mw-headline">Characters and escaping</span></h2>
<p>XML documents consist entirely of characters from the&nbsp;<a title="Unicode" href="https://en.wikipedia.org/wiki/Unicode">Unicode</a>&nbsp;repertoire. Except for a small number of specifically excluded&nbsp;<a class="mw-redirect" title="Control characters" href="https://en.wikipedia.org/wiki/Control_characters">control characters</a>, any character defined by Unicode may appear within the content of an XML document.</p>
<p>XML includes facilities for identifying the&nbsp;<em>encoding</em>&nbsp;of the Unicode characters that make up the document, and for expressing characters that, for one reason or another, cannot be used directly.</p>
<h3><span id="Valid_characters" class="mw-headline">Valid characters</span></h3>
<div class="hatnote navigation-not-searchable">Main article:&nbsp;<a title="Valid characters in XML" href="https://en.wikipedia.org/wiki/Valid_characters_in_XML">Valid characters in XML</a></div>
<p>Unicode code points in the following ranges are valid in XML 1.0 documents:</p>
<ul>
<li>U+0009 (Horizontal Tab), U+000A (Line Feed), U+000D (Carriage Return): these are the only&nbsp;<a title="C0 and C1 control codes" href="https://en.wikipedia.org/wiki/C0_and_C1_control_codes">C0</a>&nbsp;controls accepted in XML 1.0;</li>
<li>U+0020&ndash;U+D7FF, U+E000&ndash;U+FFFD: this excludes some non-characters in the&nbsp;<a class="mw-redirect" title="Basic Multilingual Plane" href="https://en.wikipedia.org/wiki/Basic_Multilingual_Plane">BMP</a>&nbsp;(all surrogates, U+FFFE and U+FFFF are forbidden);</li>
<li>U+10000&ndash;U+10FFFF: this includes all code points in supplementary planes, including non-characters.</li>
</ul>
<p>XML 1.1 extends the set of allowed characters to include all the above, plus the remaining characters in the range U+0001&ndash;U+001F.&nbsp;At the same time, however, it restricts the use of C0 and&nbsp;<a title="C0 and C1 control codes" href="https://en.wikipedia.org/wiki/C0_and_C1_control_codes">C1</a>&nbsp;control characters other than U+0009 (Horizontal Tab), U+000A (Line Feed), U+000D (Carriage Return), and U+0085 (Next Line) by requiring them to be written in escaped form (for example U+0001 must be written as&nbsp;<code>&amp;#x01;</code>&nbsp;or its equivalent). In the case of C1 characters, this restriction is a backwards incompatibility; it was introduced to allow common encoding errors to be detected.</p>
<p>The code point&nbsp;<a title="Null character" href="https://en.wikipedia.org/wiki/Null_character">U+0000</a>&nbsp;(Null) is the only character that is not permitted in any XML 1.0 or 1.1 document.</p>
<h3><span id="Encoding_detection" class="mw-headline">Encoding detection</span></h3>
<p>The Unicode character set can be encoded into bytes for storage or transmission in a variety of different ways, called "encodings". Unicode itself defines encodings that cover the entire repertoire; well-known ones include&nbsp;<a title="UTF-8" href="https://en.wikipedia.org/wiki/UTF-8">UTF-8</a>&nbsp;and&nbsp;<a title="UTF-16" href="https://en.wikipedia.org/wiki/UTF-16">UTF-16</a>.&nbsp;There are many other text encodings that predate Unicode, such as&nbsp;<a title="ASCII" href="https://en.wikipedia.org/wiki/ASCII">ASCII</a>&nbsp;and&nbsp;<a title="ISO/IEC 8859" href="https://en.wikipedia.org/wiki/ISO/IEC_8859">ISO/IEC 8859</a>; their character repertoires in almost every case are subsets of the Unicode character set.</p>
<p>XML allows the use of any of the Unicode-defined encodings, and any other encodings whose characters also appear in Unicode. XML also provides a mechanism whereby an XML processor can reliably, without any prior knowledge, determine which encoding is being used.&nbsp;Encodings other than UTF-8 and UTF-16 are not necessarily recognized by every XML parser.</p>
<h3><span id="Escaping" class="mw-headline">Escaping</span></h3>
<p>XML provides&nbsp;<em><a title="Escape sequence" href="https://en.wikipedia.org/wiki/Escape_sequence">escape</a></em>&nbsp;facilities for including characters that are problematic to include directly. For example:</p>
<ul>
<li>The characters "&lt;" and "&amp;" are key syntax markers and may never appear in content outside a&nbsp;<a title="CDATA" href="https://en.wikipedia.org/wiki/CDATA">CDATA</a>&nbsp;section. It is allowed, but not recommended, to use "&lt;" in XML entity values.</li>
<li>Some character encodings support only a subset of Unicode. For example, it is legal to encode an XML document in ASCII, but ASCII lacks code points for Unicode characters such as "&eacute;".</li>
<li>It might not be possible to type the character on the author's machine.</li>
<li>Some characters have&nbsp;<a title="Homoglyph" href="https://en.wikipedia.org/wiki/Homoglyph">glyphs</a>&nbsp;that cannot be visually distinguished from other characters, such as the&nbsp;<a title="Non-breaking space" href="https://en.wikipedia.org/wiki/Non-breaking_space">non-breaking space</a>&nbsp;(<code>&amp;#xa0;</code>) " " and the&nbsp;<a title="Space (punctuation)" href="https://en.wikipedia.org/wiki/Space_(punctuation)">space</a>&nbsp;(<code>&amp;#x20;</code>) " ", and the&nbsp;<a class="mw-redirect" title="А" href="https://en.wikipedia.org/wiki/%D0%90">Cyrillic capital letter A</a>&nbsp;(<code>&amp;#x410;</code>) "А" and the&nbsp;<a title="A" href="https://en.wikipedia.org/wiki/A">Latin capital letter A</a>&nbsp;(<code>&amp;#x41;</code>) "A".</li>
</ul>
<p>There are five&nbsp;<a title="List of XML and HTML character entity references" href="https://en.wikipedia.org/wiki/List_of_XML_and_HTML_character_entity_references#Predefined_entities_in_XML">predefined entities</a>:</p>
<ul>
<li><code>&amp;lt;</code>&nbsp;represents "&lt;";</li>
<li><code>&amp;gt;</code>&nbsp;represents "&gt;";</li>
<li><code>&amp;amp;</code>&nbsp;represents "&amp;";</li>
<li><code>&amp;apos;</code>&nbsp;represents "<span class="monospaced">'</span>";</li>
<li><code>&amp;quot;</code>&nbsp;represents '<span class="monospaced">"</span>'.</li>
</ul>
<p>All permitted Unicode characters may be represented with a&nbsp;<em><a title="Numeric character reference" href="https://en.wikipedia.org/wiki/Numeric_character_reference">numeric character reference</a></em>. Consider the Chinese character "中", whose numeric code in Unicode is hexadecimal 4E2D, or decimal 20,013. A user whose keyboard offers no method for entering this character could still insert it in an XML document encoded either as&nbsp;<code>&amp;#20013;</code>&nbsp;or&nbsp;<code>&amp;#x4e2d;</code>. Similarly, the string "I &lt;3 J&ouml;rg" could be encoded for inclusion in an XML document as&nbsp;<code>I &amp;lt;3 J&amp;#xF6;rg</code>.</p>
<p><code>&amp;#0;</code>&nbsp;is not permitted, however, because the&nbsp;<a title="Null character" href="https://en.wikipedia.org/wiki/Null_character">null character</a>&nbsp;is one of the control characters excluded from XML, even when using a numeric character reference.<sup id="cite_ref-17" class="reference"><a href="https://en.wikipedia.org/wiki/XML#cite_note-17">[15]</a></sup>&nbsp;An alternative encoding mechanism such as&nbsp;<a title="Base64" href="https://en.wikipedia.org/wiki/Base64">Base64</a>&nbsp;is needed to represent such characters.</p>
<h3><span id="Comments" class="mw-headline">Comments</span></h3>
<p>Comments may appear anywhere in a document outside other markup. Comments cannot appear before the XML declaration. Comments begin with&nbsp;<code>&lt;!--</code>&nbsp;and end with&nbsp;<code>--&gt;</code>. For compatibility with&nbsp;<a class="mw-redirect" title="SGML" href="https://en.wikipedia.org/wiki/SGML">SGML</a>, the string "--" (double-hyphen) is not allowed inside comments;&nbsp;this means comments cannot be nested. The ampersand has no special significance within comments, so entity and character references are not recognized as such, and there is no way to represent characters outside the character set of the document encoding.</p>
<p>An example of a valid comment:&nbsp;<code>&lt;!--no need to escape &lt;code&gt; &amp; such in comments--&gt;</code></p>

<h2><span id="Syntactical_correctness_and_error-handling" class="mw-headline">Syntactical correctness and error-handling</span></h2>
<div class="hatnote navigation-not-searchable">Main article:&nbsp;<a title="Well-formed document" href="https://en.wikipedia.org/wiki/Well-formed_document">Well-formed document</a></div>
<p>The XML specification defines an XML document as a&nbsp;<a title="Well-formed element" href="https://en.wikipedia.org/wiki/Well-formed_element">well-formed</a>&nbsp;text, meaning that it satisfies a list of syntax rules provided in the specification. Some key points in the fairly lengthy list include:</p>
<ul>
<li>The document contains only properly encoded legal Unicode characters.</li>
<li>None of the special syntax characters such as&nbsp;<code>&lt;</code>&nbsp;and&nbsp;<code>&amp;</code>&nbsp;appear except when performing their markup-delineation roles.</li>
<li>The start-tag, end-tag, and empty-element tag that delimit elements are correctly nested, with&nbsp;<a title="Tag omission" href="https://en.wikipedia.org/wiki/Tag_omission">none missing</a>&nbsp;and none overlapping.</li>
<li>Tag names are case-sensitive; the start-tag and end-tag must match exactly.</li>
<li>Tag names cannot contain any of the characters&nbsp;!"#$%&amp;'()*+,/;&lt;=&gt;?@[\]^`{|}~, nor a space character, and cannot begin with "-", ".", or a numeric digit.</li>
<li>A single root element contains all the other elements.</li>
</ul>
<p>The definition of an XML document excludes texts that contain violations of well-formedness rules; they are simply not XML. An XML processor that encounters such a violation is required to report such errors and to cease normal processing. This policy, occasionally referred to as "<a title="Draco (lawgiver)" href="https://en.wikipedia.org/wiki/Draco_(lawgiver)">draconian</a>&nbsp;error handling," stands in notable contrast to the behavior of programs that process&nbsp;<a title="HTML" href="https://en.wikipedia.org/wiki/HTML">HTML</a>, which are designed to produce a reasonable result even in the presence of severe markup errors.&nbsp;XML's policy in this area has been criticized as a violation of&nbsp;<a class="mw-redirect" title="Postel's law" href="https://en.wikipedia.org/wiki/Postel%27s_law">Postel's law</a>&nbsp;("Be conservative in what you send; be liberal in what you accept").</p>
<p>The XML specification defines a&nbsp;<a title="XML validation" href="https://en.wikipedia.org/wiki/XML_validation">valid XML document</a>&nbsp;as a&nbsp;<a class="mw-redirect" title="Well-formed XML document" href="https://en.wikipedia.org/wiki/Well-formed_XML_document">well-formed XML document</a>&nbsp;which also conforms to the rules of a&nbsp;<a class="mw-redirect" title="Document Type Definition" href="https://en.wikipedia.org/wiki/Document_Type_Definition">Document Type Definition</a>&nbsp;(DTD).</p>
<h2><span id="Schemas_and_validation" class="mw-headline">Schemas and validation</span></h2>
<p>In addition to being well-formed, an XML document may be&nbsp;<em>valid</em>. This means that it contains a reference to a&nbsp;<a class="mw-redirect" title="Document Type Definition" href="https://en.wikipedia.org/wiki/Document_Type_Definition">Document Type Definition</a>&nbsp;(DTD), and that its elements and attributes are declared in that DTD and follow the grammatical rules for them that the DTD specifies.</p>
<p>XML processors are classified as&nbsp;<em>validating</em>&nbsp;or&nbsp;<em>non-validating</em>&nbsp;depending on whether or not they check XML documents for validity. A processor that discovers a validity error must be able to report it, but may continue normal processing.</p>
<p>A DTD is an example of a&nbsp;<em><a title="XML schema" href="https://en.wikipedia.org/wiki/XML_schema">schema</a></em>&nbsp;or&nbsp;<em>grammar</em>. Since the initial publication of XML 1.0, there has been substantial work in the area of schema languages for XML. Such schema languages typically constrain the set of elements that may be used in a document, which attributes may be applied to them, the order in which they may appear, and the allowable parent/child relationships.</p>
<h3><span id="Document_type_definition" class="mw-headline">Document type definition</span></h3>
<div class="hatnote navigation-not-searchable">Main article:&nbsp;<a title="Document type definition" href="https://en.wikipedia.org/wiki/Document_type_definition">Document type definition</a></div>
<p>The oldest schema language for XML is the&nbsp;<a title="Document type definition" href="https://en.wikipedia.org/wiki/Document_type_definition">document type definition</a>&nbsp;(DTD), inherited from SGML.</p>
<p>DTDs have the following benefits:</p>
<ul>
<li>DTD support is ubiquitous due to its inclusion in the XML 1.0 standard.</li>
<li>DTDs are terse compared to element-based schema languages and consequently present more information in a single screen.</li>
<li>DTDs allow the declaration of&nbsp;<a title="SGML entity" href="https://en.wikipedia.org/wiki/SGML_entity">standard public entity sets</a>&nbsp;for publishing characters.</li>
<li>DTDs define a&nbsp;<em>document type</em>&nbsp;rather than the types used by a namespace, thus grouping all constraints for a document in a single collection.</li>
</ul>
<p>DTDs have the following limitations:</p>
<ul>
<li>They have no explicit support for newer&nbsp;<a class="mw-redirect" title="Feature (software design)" href="https://en.wikipedia.org/wiki/Feature_(software_design)">features</a>&nbsp;of XML, most importantly&nbsp;<a class="mw-redirect" title="XML Namespace" href="https://en.wikipedia.org/wiki/XML_Namespace">namespaces</a>.</li>
<li>They lack expressiveness. XML DTDs are simpler than SGML DTDs and there are certain structures that cannot be expressed with regular grammars. DTDs only support rudimentary datatypes.</li>
<li>They lack readability. DTD designers typically make heavy use of parameter entities (which behave essentially as textual&nbsp;<a title="Macro (computer science)" href="https://en.wikipedia.org/wiki/Macro_(computer_science)">macros</a>), which make it easier to define complex grammars, but at the expense of clarity.</li>
<li>They use a syntax based on&nbsp;<a title="Regular expression" href="https://en.wikipedia.org/wiki/Regular_expression">regular expression</a>&nbsp;syntax, inherited from SGML, to describe the schema. Typical XML APIs such as&nbsp;<a title="Simple API for XML" href="https://en.wikipedia.org/wiki/Simple_API_for_XML">SAX</a>&nbsp;do not attempt to offer applications a structured representation of the syntax, so it is less accessible to programmers than an element-based syntax may be.</li>
</ul>
<p>Two peculiar features that distinguish DTDs from other schema types are the syntactic support for embedding a DTD within XML documents and for defining&nbsp;<em>entities</em>, which are arbitrary fragments of text or markup that the XML processor inserts in the DTD itself and in the XML document wherever they are referenced, like character escapes.</p>
<p>DTD technology is still used in many applications because of its ubiquity.</p>
<h3><span id="Schema" class="mw-headline">Schema</span></h3>
<div class="hatnote navigation-not-searchable">Main article:&nbsp;<a title="XML Schema (W3C)" href="https://en.wikipedia.org/wiki/XML_Schema_(W3C)">XML Schema (W3C)</a></div>
<p>A newer schema language, described by the W3C as the successor of DTDs, is&nbsp;<a title="XML Schema (W3C)" href="https://en.wikipedia.org/wiki/XML_Schema_(W3C)">XML Schema</a>, often referred to by the&nbsp;<a class="mw-redirect" title="Initialism" href="https://en.wikipedia.org/wiki/Initialism">initialism</a>&nbsp;for XML Schema instances, XSD (XML Schema Definition). XSDs are far more powerful than DTDs in describing XML languages. They use a rich&nbsp;<a title="Data type" href="https://en.wikipedia.org/wiki/Data_type">datatyping</a>&nbsp;system and allow for more detailed constraints on an XML document's logical structure. XSDs also use an XML-based format, which makes it possible to use ordinary XML tools to help process them.</p>

<h3><span id="RELAX_NG" class="mw-headline">RELAX NG</span></h3>
<p><a title="RELAX NG" href="https://en.wikipedia.org/wiki/RELAX_NG">RELAX NG</a>&nbsp;(Regular Language for XML Next Generation) was initially specified by&nbsp;<a title="OASIS (organization)" href="https://en.wikipedia.org/wiki/OASIS_(organization)">OASIS</a>&nbsp;and is now a standard (Part 2:&nbsp;<em>Regular-grammar-based validation</em>&nbsp;of&nbsp;<a class="mw-redirect" title="DSDL" href="https://en.wikipedia.org/wiki/DSDL">ISO/IEC 19757 - DSDL</a>). RELAX NG schemas may be written in either an XML based syntax or a more compact non-XML syntax; the two syntaxes are&nbsp;<a class="mw-redirect" title="Isomorphic" href="https://en.wikipedia.org/wiki/Isomorphic">isomorphic</a>&nbsp;and&nbsp;<a title="James Clark (programmer)" href="https://en.wikipedia.org/wiki/James_Clark_(programmer)">James Clark</a>'s conversion tool&mdash;<a class="external text" href="http://www.thaiopensource.com/relaxng/trang.html" rel="nofollow">Trang</a>&mdash;can convert between them without loss of information. RELAX NG has a simpler definition and validation framework than XML Schema, making it easier to use and implement. It also has the ability to use&nbsp;<a class="mw-redirect" title="Datatype" href="https://en.wikipedia.org/wiki/Datatype">datatype</a>&nbsp;framework&nbsp;<a title="Plug-in (computing)" href="https://en.wikipedia.org/wiki/Plug-in_(computing)">plug-ins</a>; a RELAX NG schema author, for example, can require values in an XML document to conform to definitions in XML Schema Datatypes.</p>
<h3><span id="Schematron" class="mw-headline">Schematron</span></h3>
<p><a title="Schematron" href="https://en.wikipedia.org/wiki/Schematron">Schematron</a>&nbsp;is a language for making&nbsp;<a class="mw-redirect" title="Assertion (computing)" href="https://en.wikipedia.org/wiki/Assertion_(computing)">assertions</a>&nbsp;about the presence or absence of patterns in an XML document. It typically uses&nbsp;<a title="XPath" href="https://en.wikipedia.org/wiki/XPath">XPath</a>&nbsp;expressions. Schematron is now a standard (Part 3:&nbsp;<em>Rule-based validation</em>&nbsp;of&nbsp;<a class="mw-redirect" title="DSDL" href="https://en.wikipedia.org/wiki/DSDL">ISO/IEC 19757 - DSDL</a>).</p>
<h3><span id="DSDL_and_other_schema_languages" class="mw-headline">DSDL and other schema languages</span></h3>
<p><a title="Document Schema Definition Languages" href="https://en.wikipedia.org/wiki/Document_Schema_Definition_Languages">DSDL</a>&nbsp;(Document Schema Definition Languages) is a multi-part ISO/IEC standard (ISO/IEC 19757) that brings together a comprehensive set of small schema languages, each targeted at specific problems. DSDL includes&nbsp;<a title="RELAX NG" href="https://en.wikipedia.org/wiki/RELAX_NG">RELAX NG</a>&nbsp;full and compact syntax,&nbsp;<a title="Schematron" href="https://en.wikipedia.org/wiki/Schematron">Schematron</a>&nbsp;assertion language, and languages for defining datatypes, character repertoire constraints, renaming and entity expansion, and namespace-based&nbsp;<a title="Routing" href="https://en.wikipedia.org/wiki/Routing">routing</a>&nbsp;of document fragments to different validators. DSDL schema languages do not have the vendor support of XML Schemas yet, and are to some extent a grassroots reaction of industrial publishers to the lack of utility of XML Schemas for&nbsp;<a title="Publishing" href="https://en.wikipedia.org/wiki/Publishing">publishing</a>.</p>
<p>Some schema languages not only describe the structure of a particular XML format but also offer limited facilities to influence processing of individual XML files that conform to this format. DTDs and XSDs both have this ability; they can for instance provide the&nbsp;<a class="mw-redirect" title="Infoset" href="https://en.wikipedia.org/wiki/Infoset">infoset</a>&nbsp;augmentation facility and attribute defaults. RELAX NG and Schematron intentionally do not provide these.</p>
<h2><span id="Related_specifications" class="mw-headline">Related specifications</span></h2>
<p>A cluster of specifications closely related to XML have been developed, starting soon after the initial publication of XML 1.0. It is frequently the case that the term "XML" is used to refer to XML together with one or more of these other technologies that have come to be seen as part of the XML core.</p>
<ul>
<li><a title="XML namespace" href="https://en.wikipedia.org/wiki/XML_namespace">XML namespaces</a>&nbsp;enable the same document to contain XML elements and attributes taken from different vocabularies, without any&nbsp;<a title="Naming collision" href="https://en.wikipedia.org/wiki/Naming_collision">naming collisions</a>&nbsp;occurring. Although XML Namespaces are not part of the XML specification itself, virtually all XML software also supports XML Namespaces.</li>
<li><a title="XML Base" href="https://en.wikipedia.org/wiki/XML_Base">XML Base</a>&nbsp;defines the&nbsp;<code>xml:base</code>&nbsp;attribute, which may be used to set the base for resolution of relative URI references within the scope of a single XML element.</li>
<li><a title="XML Information Set" href="https://en.wikipedia.org/wiki/XML_Information_Set">XML Information Set</a>&nbsp;or XML Infoset is an abstract data model for XML documents in terms of&nbsp;<em>information items</em>. The infoset is commonly used in the specifications of XML languages, for convenience in describing constraints on the XML constructs those languages allow.</li>
<li><a title="XSL" href="https://en.wikipedia.org/wiki/XSL">XSL</a>&nbsp;(Extensible Stylesheet Language) is a family of languages used to transform and render XML documents, split into three parts:</li>
</ul>
<dl>
<dd>
<ul>
<li><a title="XSLT" href="https://en.wikipedia.org/wiki/XSLT">XSLT</a>&nbsp;(XSL Transformations), an XML language for transforming XML documents into other XML documents or other formats such as HTML, plain text, or XSL-FO. XSLT is very tightly coupled with XPath, which it uses to address components of the input XML document, mainly elements and attributes.</li>
<li><a class="mw-redirect" title="XSL-FO" href="https://en.wikipedia.org/wiki/XSL-FO">XSL-FO</a>&nbsp;(XSL Formatting Objects), an XML language for rendering XML documents, often used to generate PDFs.</li>
<li><a title="XPath" href="https://en.wikipedia.org/wiki/XPath">XPath</a>&nbsp;(XML Path Language), a non-XML language for addressing the components (elements, attributes, and so on) of an XML document. XPath is widely used in other core-XML specifications and in programming libraries for accessing XML-encoded data.</li>
</ul>
</dd>
</dl>
<ul>
<li><a title="XQuery" href="https://en.wikipedia.org/wiki/XQuery">XQuery</a>&nbsp;(XML Query) is an XML query language strongly rooted in XPath and XML Schema. It provides methods to access, manipulate and return XML, and is mainly conceived as a query language for&nbsp;<a title="XML database" href="https://en.wikipedia.org/wiki/XML_database">XML databases</a>.</li>
<li><a title="XML Signature" href="https://en.wikipedia.org/wiki/XML_Signature">XML Signature</a>&nbsp;defines syntax and processing rules for creating&nbsp;<a title="Digital signature" href="https://en.wikipedia.org/wiki/Digital_signature">digital signatures</a>&nbsp;on XML content.</li>
<li><a title="XML Encryption" href="https://en.wikipedia.org/wiki/XML_Encryption">XML Encryption</a>&nbsp;defines syntax and processing rules for&nbsp;<a title="Encryption" href="https://en.wikipedia.org/wiki/Encryption">encrypting</a>&nbsp;XML content.</li>
<li><a class="new" title="Xml-model (page does not exist)" href="https://en.wikipedia.org/w/index.php?title=Xml-model&amp;action=edit&amp;redlink=1">xml-model</a>&nbsp;(Part 11:&nbsp;<em>Schema Association</em>&nbsp;of&nbsp;<a class="mw-redirect" title="DSDL" href="https://en.wikipedia.org/wiki/DSDL">ISO/IEC 19757 - DSDL</a>) defines a means of associating any xml document with any of the schema types mentioned&nbsp;<a href="https://en.wikipedia.org/wiki/XML#Schemas_and_validation">above</a>.</li>
</ul>
<p>Some other specifications conceived as part of the "XML Core" have failed to find wide adoption, including&nbsp;<a title="XInclude" href="https://en.wikipedia.org/wiki/XInclude">XInclude</a>,&nbsp;<a title="XLink" href="https://en.wikipedia.org/wiki/XLink">XLink</a>, and&nbsp;<a title="XPointer" href="https://en.wikipedia.org/wiki/XPointer">XPointer</a>.</p>
<h2><span id="Programming_interfaces" class="mw-headline">Programming interfaces</span></h2>
<p>The design goals of XML include, "It shall be easy to write programs which process XML documents."<sup id="cite_ref-XML_Goals_6-1" class="reference"><a href="https://en.wikipedia.org/wiki/XML#cite_note-XML_Goals-6">[6]</a></sup>&nbsp;Despite this, the XML specification contains almost no information about how programmers might go about doing such processing. The&nbsp;<a class="mw-redirect" title="XML Infoset" href="https://en.wikipedia.org/wiki/XML_Infoset">XML Infoset</a>&nbsp;specification provides a vocabulary to refer to the constructs within an XML document, but does not provide any guidance on how to access this information. A variety of&nbsp;<a class="mw-redirect" title="API" href="https://en.wikipedia.org/wiki/API">APIs</a>&nbsp;for accessing XML have been developed and used, and some have been standardized.</p>
<p>Existing APIs for XML processing tend to fall into these categories:</p>
<ul>
<li>Stream-oriented APIs accessible from a programming language, for example&nbsp;<a title="Simple API for XML" href="https://en.wikipedia.org/wiki/Simple_API_for_XML">SAX</a>&nbsp;and&nbsp;<a title="StAX" href="https://en.wikipedia.org/wiki/StAX">StAX</a>.</li>
<li>Tree-traversal APIs accessible from a programming language, for example&nbsp;<a class="mw-redirect" title="DOM (XML API)" href="https://en.wikipedia.org/wiki/DOM_(XML_API)">DOM</a>.</li>
<li><a title="XML data binding" href="https://en.wikipedia.org/wiki/XML_data_binding">XML data binding</a>, which provides an automated translation between an XML document and programming-language objects.</li>
<li>Declarative transformation languages such as&nbsp;<a title="XSLT" href="https://en.wikipedia.org/wiki/XSLT">XSLT</a>&nbsp;and&nbsp;<a title="XQuery" href="https://en.wikipedia.org/wiki/XQuery">XQuery</a>.</li>
<li>Syntax extensions to general-purpose programming languages, for example&nbsp;<a class="mw-redirect" title="LINQ" href="https://en.wikipedia.org/wiki/LINQ">LINQ</a>&nbsp;and&nbsp;<a title="Scala (programming language)" href="https://en.wikipedia.org/wiki/Scala_(programming_language)">Scala</a>.</li>
</ul>
<p>Stream-oriented facilities require less memory and, for certain tasks based on a linear traversal of an XML document, are faster and simpler than other alternatives. Tree-traversal and data-binding APIs typically require the use of much more memory, but are often found more convenient for use by programmers; some include declarative retrieval of document components via the use of XPath expressions.</p>
<p>XSLT is designed for declarative description of XML document transformations, and has been widely implemented both in server-side packages and Web browsers. XQuery overlaps XSLT in its functionality, but is designed more for searching of large&nbsp;<a title="XML database" href="https://en.wikipedia.org/wiki/XML_database">XML databases</a>.</p>
<h3><span id="Simple_API_for_XML" class="mw-headline">Simple API for XML</span></h3>
<div class="hatnote navigation-not-searchable">Main article:&nbsp;<a title="Simple API for XML" href="https://en.wikipedia.org/wiki/Simple_API_for_XML">Simple API for XML</a></div>
<p><a title="Simple API for XML" href="https://en.wikipedia.org/wiki/Simple_API_for_XML">Simple API for XML</a>&nbsp;(SAX) is a&nbsp;<a title="Lexical analysis" href="https://en.wikipedia.org/wiki/Lexical_analysis">lexical</a>,&nbsp;<a title="Event-driven programming" href="https://en.wikipedia.org/wiki/Event-driven_programming">event-driven</a>&nbsp;API in which a document is read serially and its contents are reported as&nbsp;<a title="Callback (computer programming)" href="https://en.wikipedia.org/wiki/Callback_(computer_programming)">callbacks</a>&nbsp;to various&nbsp;<a class="mw-redirect" title="Method (computer science)" href="https://en.wikipedia.org/wiki/Method_(computer_science)">methods</a>&nbsp;on a&nbsp;<a class="mw-redirect" title="Event handler" href="https://en.wikipedia.org/wiki/Event_handler">handler object</a>&nbsp;of the user's design. SAX is fast and efficient to implement, but difficult to use for extracting information at random from the XML, since it tends to burden the application author with keeping track of what part of the document is being processed. It is better suited to situations in which certain types of information are always handled the same way, no matter where they occur in the document.</p>
<h3><span id="Pull_parsing" class="mw-headline">Pull parsing</span></h3>
<p>Pull parsing treats the document as a series of items read in sequence using the&nbsp;<a title="Iterator pattern" href="https://en.wikipedia.org/wiki/Iterator_pattern">iterator design pattern</a>. This allows for writing of&nbsp;<a title="Recursive descent parser" href="https://en.wikipedia.org/wiki/Recursive_descent_parser">recursive descent parsers</a>&nbsp;in which the structure of the code performing the parsing mirrors the structure of the XML being parsed, and intermediate parsed results can be used and accessed as local variables within the functions performing the parsing, or passed down (as function parameters) into lower-level functions, or returned (as function return values) to higher-level functions.<sup id="cite_ref-23" class="reference"><a href="https://en.wikipedia.org/wiki/XML#cite_note-23">[21]</a></sup>&nbsp;Examples of pull parsers include Data::Edit::Xml in&nbsp;<a class="mw-redirect" title="Perl (programming language)" href="https://en.wikipedia.org/wiki/Perl_(programming_language)">Perl</a>,&nbsp;<a title="StAX" href="https://en.wikipedia.org/wiki/StAX">StAX</a>&nbsp;in the&nbsp;<a title="Java (programming language)" href="https://en.wikipedia.org/wiki/Java_(programming_language)">Java</a>&nbsp;programming language, XMLPullParser in&nbsp;<a title="Smalltalk" href="https://en.wikipedia.org/wiki/Smalltalk">Smalltalk</a>, XMLReader in&nbsp;<a title="PHP" href="https://en.wikipedia.org/wiki/PHP">PHP</a>, ElementTree.iterparse in&nbsp;<a title="Python (programming language)" href="https://en.wikipedia.org/wiki/Python_(programming_language)">Python</a>, System.Xml.XmlReader in the&nbsp;<a title=".NET Framework" href="https://en.wikipedia.org/wiki/.NET_Framework">.NET Framework</a>, and the DOM traversal API (NodeIterator and TreeWalker).</p>
<p>A pull parser creates an iterator that sequentially visits the various elements, attributes, and data in an XML document. Code that uses this iterator can test the current item (to tell, for example, whether it is a start-tag or end-tag, or text), and inspect its attributes (local name,&nbsp;<a title="XML namespace" href="https://en.wikipedia.org/wiki/XML_namespace">namespace</a>, values of XML attributes, value of text, etc.), and can also move the iterator to the next item. The code can thus extract information from the document as it traverses it. The recursive-descent approach tends to lend itself to keeping data as typed local variables in the code doing the parsing, while SAX, for instance, typically requires a parser to manually maintain intermediate data within a stack of elements that are parent elements of the element being parsed. Pull-parsing code can be more straightforward to understand and maintain than SAX parsing code.</p>
<h3><span id="Document_Object_Model" class="mw-headline">Document Object Model</span></h3>
<div class="hatnote navigation-not-searchable">Main article:&nbsp;<a title="Document Object Model" href="https://en.wikipedia.org/wiki/Document_Object_Model">Document Object Model</a></div>
<p><a title="Document Object Model" href="https://en.wikipedia.org/wiki/Document_Object_Model">Document Object Model</a>&nbsp;(DOM) is an API that allows for navigation of the entire document as if it were a tree of&nbsp;<a title="Node (computer science)" href="https://en.wikipedia.org/wiki/Node_(computer_science)">node</a>&nbsp;<a title="Object (computer science)" href="https://en.wikipedia.org/wiki/Object_(computer_science)">objects</a>&nbsp;representing the document's contents. A DOM document can be created by a parser, or can be generated manually by users (with limitations). Data types in DOM nodes are abstract; implementations provide their own programming language-specific&nbsp;<a title="Language binding" href="https://en.wikipedia.org/wiki/Language_binding">bindings</a>. DOM implementations tend to be&nbsp;<a title="Memory" href="https://en.wikipedia.org/wiki/Memory">memory</a>&nbsp;intensive, as they generally require the entire document to be loaded into memory and constructed as a tree of objects before access is allowed.</p>
<h3><span id="Data_binding" class="mw-headline">Data binding</span></h3>
<p><a title="XML data binding" href="https://en.wikipedia.org/wiki/XML_data_binding">XML data binding</a>&nbsp;is the binding of XML documents to a hierarchy of custom and strongly typed objects, in contrast to the generic objects created by a DOM parser. This approach simplifies code development, and in many cases allows problems to be identified at compile time rather than run-time. It is suitable for applications where the document structure is known and fixed at the time the application is written. Example data binding systems include the&nbsp;<a title="Java Architecture for XML Binding" href="https://en.wikipedia.org/wiki/Java_Architecture_for_XML_Binding">Java Architecture for XML Binding</a>&nbsp;(JAXB), XML Serialization in&nbsp;<a title=".NET Framework" href="https://en.wikipedia.org/wiki/.NET_Framework">.NET Framework</a>.<sup id="cite_ref-24" class="reference"><a href="https://en.wikipedia.org/wiki/XML#cite_note-24">[22]</a></sup>&nbsp;and XML serialization in&nbsp;<a title="GSOAP" href="https://en.wikipedia.org/wiki/GSOAP">gSOAP</a>.</p>
<h3><span id="XML_as_data_type" class="mw-headline">XML as data type</span></h3>
<p>XML has appeared as a&nbsp;<a class="mw-redirect" title="First-class data type" href="https://en.wikipedia.org/wiki/First-class_data_type">first-class data type</a>&nbsp;in other languages. The&nbsp;<a class="mw-redirect" title="E4X" href="https://en.wikipedia.org/wiki/E4X">ECMAScript for XML</a>&nbsp;(E4X) extension to the&nbsp;<a title="ECMAScript" href="https://en.wikipedia.org/wiki/ECMAScript">ECMAScript</a>/JavaScript language explicitly defines two specific objects (XML and XMLList) for JavaScript, which support XML document nodes and XML node lists as distinct objects and use a dot-notation specifying parent-child relationships.&nbsp;E4X is supported by the&nbsp;<a title="Mozilla" href="https://en.wikipedia.org/wiki/Mozilla">Mozilla</a>&nbsp;2.5+ browsers (though now deprecated) and Adobe&nbsp;<a class="mw-redirect" title="Actionscript" href="https://en.wikipedia.org/wiki/Actionscript">Actionscript</a>, but has not been adopted more universally. Similar notations are used in Microsoft's&nbsp;<a class="mw-redirect" title="LINQ" href="https://en.wikipedia.org/wiki/LINQ">LINQ</a>&nbsp;implementation for Microsoft .NET 3.5 and above, and in&nbsp;<a title="Scala (programming language)" href="https://en.wikipedia.org/wiki/Scala_(programming_language)">Scala</a>&nbsp;(which uses the Java VM). The open-source xmlsh application, which provides a Linux-like shell with special features for XML manipulation, similarly treats XML as a data type, using the &lt;[ ]&gt; notation.&nbsp;The&nbsp;<a title="Resource Description Framework" href="https://en.wikipedia.org/wiki/Resource_Description_Framework">Resource Description Framework</a>&nbsp;defines a data type&nbsp;<code>rdf:XMLLiteral</code>&nbsp;to hold wrapped,&nbsp;<a title="Canonical XML" href="https://en.wikipedia.org/wiki/Canonical_XML">canonical XML</a>.&nbsp;Facebook has produced extensions to the&nbsp;<a title="PHP" href="https://en.wikipedia.org/wiki/PHP">PHP</a>&nbsp;and&nbsp;<a title="JavaScript" href="https://en.wikipedia.org/wiki/JavaScript">JavaScript</a>&nbsp;languages that add XML to the core syntax in a similar fashion to E4X, namely&nbsp;<a title="XHP" href="https://en.wikipedia.org/wiki/XHP">XHP</a>&nbsp;and&nbsp;<a class="mw-redirect" title="React (JavaScript library)" href="https://en.wikipedia.org/wiki/React_(JavaScript_library)#JSX">JSX</a>&nbsp;respectively.</p>
<h2><span id="History" class="mw-headline">History</span></h2>
<p>XML is an application&nbsp;<a title="Profile (engineering)" href="https://en.wikipedia.org/wiki/Profile_(engineering)">profile</a>&nbsp;of SGML (ISO 8879).</p>
<p>The versatility of SGML for dynamic information display was understood by early digital media publishers in the late 1980s prior to the rise of the Internet.&nbsp;By the mid-1990s some practitioners of SGML had gained experience with the then-new&nbsp;<a title="World Wide Web" href="https://en.wikipedia.org/wiki/World_Wide_Web">World Wide Web</a>, and believed that SGML offered solutions to some of the problems the Web was likely to face as it grew.&nbsp;<a title="Dan Connolly (computer scientist)" href="https://en.wikipedia.org/wiki/Dan_Connolly_(computer_scientist)">Dan Connolly</a>&nbsp;added SGML to the list of W3C's activities when he joined the staff in 1995; work began in mid-1996 when&nbsp;<a title="Sun Microsystems" href="https://en.wikipedia.org/wiki/Sun_Microsystems">Sun Microsystems</a>&nbsp;engineer&nbsp;<a title="Jon Bosak" href="https://en.wikipedia.org/wiki/Jon_Bosak">Jon Bosak</a>&nbsp;developed a charter and recruited collaborators. Bosak was well connected in the small community of people who had experience both in SGML and the Web.</p>
<p>XML was compiled by a&nbsp;<a title="Working group" href="https://en.wikipedia.org/wiki/Working_group">working group</a>&nbsp;of eleven members,&nbsp;supported by a (roughly) 150-member Interest Group. Technical debate took place on the Interest Group mailing list and issues were resolved by consensus or, when that failed, majority vote of the Working Group. A record of design decisions and their rationales was compiled by&nbsp;<a title="Michael Sperberg-McQueen" href="https://en.wikipedia.org/wiki/Michael_Sperberg-McQueen">Michael Sperberg-McQueen</a>&nbsp;on December 4, 1997.&nbsp;<a class="mw-redirect" title="James Clark (XML expert)" href="https://en.wikipedia.org/wiki/James_Clark_(XML_expert)">James Clark</a>&nbsp;served as Technical Lead of the Working Group, notably contributing the empty-element&nbsp;<code>&lt;empty&nbsp;/&gt;</code>&nbsp;syntax and the name "XML". Other names that had been put forward for consideration included "MAGMA" (Minimal Architecture for Generalized Markup Applications), "SLIM" (Structured Language for Internet Markup) and "MGML" (Minimal Generalized Markup Language). The co-editors of the specification were originally&nbsp;<a title="Tim Bray" href="https://en.wikipedia.org/wiki/Tim_Bray">Tim Bray</a>&nbsp;and&nbsp;<a title="Michael Sperberg-McQueen" href="https://en.wikipedia.org/wiki/Michael_Sperberg-McQueen">Michael Sperberg-McQueen</a>. Halfway through the project Bray accepted a consulting engagement with&nbsp;<a class="mw-redirect" title="Netscape Communications Corporation" href="https://en.wikipedia.org/wiki/Netscape_Communications_Corporation">Netscape</a>, provoking vociferous protests from Microsoft. Bray was temporarily asked to resign the editorship. This led to intense dispute in the Working Group, eventually solved by the appointment of Microsoft's&nbsp;<a title="Jean Paoli" href="https://en.wikipedia.org/wiki/Jean_Paoli">Jean Paoli</a>&nbsp;as a third co-editor.</p>
<p>The XML Working Group never met face-to-face; the design was accomplished using a combination of email and weekly teleconferences. The major design decisions were reached in a short burst of intense work between August and November 1996,&nbsp;when the first Working Draft of an XML specification was published.&nbsp;Further design work continued through 1997, and XML 1.0 became a&nbsp;<a class="mw-redirect" title="W3C" href="https://en.wikipedia.org/wiki/W3C">W3C</a>&nbsp;Recommendation on February 10, 1998.</p>
<h3><span id="Sources" class="mw-headline">Sources</span></h3>
<p>XML is a profile of an ISO standard SGML, and most of XML comes from SGML unchanged. From SGML comes the separation of logical and physical structures (elements and entities), the availability of grammar-based validation (DTDs), the separation of data and metadata (elements and attributes), mixed content, the separation of processing from representation (<a class="mw-redirect" title="Processing instruction" href="https://en.wikipedia.org/wiki/Processing_instruction">processing instructions</a>), and the default angle-bracket syntax. Removed were the SGML declaration (XML has a fixed delimiter set and adopts&nbsp;<a title="Unicode" href="https://en.wikipedia.org/wiki/Unicode">Unicode</a>&nbsp;as the document&nbsp;<a title="Character encoding" href="https://en.wikipedia.org/wiki/Character_encoding">character set</a>).</p>
<p>Other sources of technology for XML were the&nbsp;<a title="Text Encoding Initiative" href="https://en.wikipedia.org/wiki/Text_Encoding_Initiative">TEI</a>&nbsp;(Text Encoding Initiative), which defined a profile of SGML for use as a "transfer syntax"; and&nbsp;<a title="HTML" href="https://en.wikipedia.org/wiki/HTML">HTML</a>, in which elements were synchronous with their resource, document character sets were separate from resource encoding, the&nbsp;<code>xml:lang</code>&nbsp;attribute was invented, and (like&nbsp;<a class="mw-redirect" title="HTTP" href="https://en.wikipedia.org/wiki/HTTP">HTTP</a>) metadata accompanied the resource rather than being needed at the declaration of a link. The ERCS(Extended Reference Concrete Syntax) project of the SPREAD (Standardization Project Regarding East Asian Documents) project of the ISO-related China/Japan/Korea Document Processing expert group was the basis of XML 1.0's naming rules; SPREAD also introduced hexadecimal numeric character references and the concept of references to make available all Unicode characters. To support ERCS, XML and HTML better, the SGML standard IS 8879 was revised in 1996 and 1998 with WebSGML Adaptations. The XML header followed that of ISO&nbsp;<a title="HyTime" href="https://en.wikipedia.org/wiki/HyTime">HyTime</a>.</p>
<p>Ideas that developed during discussion that are novel in XML included the algorithm for encoding detection and the encoding header, the processing instruction target, the xml:space attribute, and the new close delimiter for empty-element tags. The notion of well-formedness as opposed to validity (which enables parsing without a schema) was first formalized in XML, although it had been implemented successfully in the Electronic Book Technology "Dynatext" software;&nbsp;the software from the University of Waterloo New Oxford English Dictionary Project; the RISP LISP SGML text processor at Uniscope, Tokyo; the US Army Missile Command IADS hypertext system; Mentor Graphics Context; Interleaf and Xerox Publishing System.</p>
<h3><span id="Versions" class="mw-headline">Versions</span></h3>
<p>There are two current versions of XML. The first (<em>XML 1.0</em>) was initially defined in 1998. It has undergone minor revisions since then, without being given a new version number, and is currently in its fifth edition, as published on November 26, 2008. It is widely implemented and still recommended for general use.</p>
<p>The second (<em>XML 1.1</em>) was initially published on February 4, 2004, the same day as XML 1.0 Third Edition,&nbsp;and is currently in its second edition, as published on August 16, 2006. It contains features (some contentious) that are intended to make XML easier to use in certain cases.&nbsp;The main changes are to enable the use of line-ending characters used on&nbsp;<a title="EBCDIC" href="https://en.wikipedia.org/wiki/EBCDIC">EBCDIC</a>&nbsp;platforms, and the use of scripts and characters absent from Unicode 3.2. XML 1.1 is not very widely implemented and is recommended for use only by those who need its particular features.</p>
<p>Prior to its fifth edition release, XML 1.0 differed from XML 1.1 in having stricter requirements for characters available for use in element and attribute names and unique identifiers: in the first four editions of XML 1.0 the characters were exclusively enumerated using a specific version of the&nbsp;<a title="Unicode" href="https://en.wikipedia.org/wiki/Unicode">Unicode</a>&nbsp;standard (Unicode 2.0 to Unicode 3.2.) The fifth edition substitutes the mechanism of XML 1.1, which is more future-proof but reduces&nbsp;<a title="Redundancy (information theory)" href="https://en.wikipedia.org/wiki/Redundancy_(information_theory)">redundancy</a>. The approach taken in the fifth edition of XML 1.0 and in all editions of XML 1.1 is that only certain characters are forbidden in names, and everything else is allowed to accommodate suitable name characters in future Unicode versions. In the fifth edition, XML names may contain characters in the&nbsp;<a title="Balinese script" href="https://en.wikipedia.org/wiki/Balinese_script">Balinese</a>,&nbsp;<a title="Cham script" href="https://en.wikipedia.org/wiki/Cham_script">Cham</a>, or&nbsp;<a title="Phoenician alphabet" href="https://en.wikipedia.org/wiki/Phoenician_alphabet">Phoenician</a>&nbsp;scripts among many others added to Unicode since Unicode 3.2.</p>
<p>Almost any Unicode code point can be used in the character data and attribute values of an XML 1.0 or 1.1 document, even if the character corresponding to the code point is not defined in the current version of Unicode. In character data and attribute values, XML 1.1 allows the use of more&nbsp;<a title="Control character" href="https://en.wikipedia.org/wiki/Control_character">control characters</a>&nbsp;than XML 1.0, but, for "robustness", most of the control characters introduced in XML 1.1 must be expressed as numeric character references (and #x7F through #x9F, which had been allowed in XML 1.0, are in XML 1.1 even required to be expressed as numeric character references). Among the supported control characters in XML 1.1 are two line break codes that must be treated as whitespace. Whitespace characters are the only control codes that can be written directly.</p>
<p>There has been discussion of an XML 2.0, although no organization has announced plans for work on such a project. XML-SW (SW for&nbsp;<a title="Skunkworks project" href="https://en.wikipedia.org/wiki/Skunkworks_project">skunkworks</a>), written by one of the original developers of XML,&nbsp;contains some proposals for what an XML 2.0 might look like: elimination of DTDs from syntax, integration of&nbsp;<a title="XML namespace" href="https://en.wikipedia.org/wiki/XML_namespace">namespaces</a>,&nbsp;<a title="XML Base" href="https://en.wikipedia.org/wiki/XML_Base">XML Base</a>&nbsp;and&nbsp;<a title="XML Information Set" href="https://en.wikipedia.org/wiki/XML_Information_Set">XML Information Set</a>&nbsp;into the base standard.</p>
<p>The World Wide Web Consortium also has an XML Binary Characterization Working Group doing preliminary research into use cases and properties for a binary encoding of XML Information Set. The working group is not chartered to produce any official standards. Since XML is by definition text-based, ITU-T and ISO are using the name&nbsp;<a title="Fast Infoset" href="https://en.wikipedia.org/wiki/Fast_Infoset">Fast Infoset</a>&nbsp;for their own binary infoset to avoid confusion (see ITU-T Rec. X.891 and ISO/IEC 24824-1).</p>
<h2><span id="Criticism" class="mw-headline">Criticism</span></h2>
<p>XML and its extensions have regularly been criticized for verbosity, complexity and redundancy.&nbsp;Mapping the basic tree model of XML to&nbsp;<a title="Type system" href="https://en.wikipedia.org/wiki/Type_system">type systems</a>&nbsp;of programming languages or databases can be difficult, especially when XML is used for exchanging highly structured data between applications, which was not its primary design goal. However,&nbsp;<a title="XML data binding" href="https://en.wikipedia.org/wiki/XML_data_binding">XML data binding</a>&nbsp;systems allow applications to access XML data directly from objects representing a&nbsp;<a title="Data structure" href="https://en.wikipedia.org/wiki/Data_structure">data structure</a>&nbsp;of the data in the programming language used, which ensures&nbsp;<a title="Type safety" href="https://en.wikipedia.org/wiki/Type_safety">type safety</a>, rather than using the&nbsp;<a title="Document Object Model" href="https://en.wikipedia.org/wiki/Document_Object_Model">DOM</a>&nbsp;or&nbsp;<a title="Simple API for XML" href="https://en.wikipedia.org/wiki/Simple_API_for_XML">SAX</a>&nbsp;to retrieve data from a direct representation of the XML itself. This is accomplished by automatically creating a mapping between elements of the XML schema&nbsp;<a class="mw-redirect" title="XSD" href="https://en.wikipedia.org/wiki/XSD">XSD</a>&nbsp;of the document and members of a class to be represented in memory. Other criticisms attempt to refute the claim that XML is a&nbsp;<a class="mw-redirect" title="Self-describing" href="https://en.wikipedia.org/wiki/Self-describing">self-describing</a>&nbsp;language&nbsp;(though the XML specification itself makes no such claim).&nbsp;<a title="JSON" href="https://en.wikipedia.org/wiki/JSON">JSON</a>,&nbsp;<a title="YAML" href="https://en.wikipedia.org/wiki/YAML">YAML</a>, and&nbsp;<a title="S-expression" href="https://en.wikipedia.org/wiki/S-expression">S-Expressions</a>&nbsp;are frequently proposed as simpler alternatives (see&nbsp;<a class="mw-redirect" title="Comparison of data serialization formats" href="https://en.wikipedia.org/wiki/Comparison_of_data_serialization_formats">Comparison of data serialization formats</a>)&nbsp;that focus on representing highly structured data rather than documents, which may contain both highly structured and relatively unstructured content. However, W3C standardized XML schema specifications offer a broader range of structured&nbsp;<a class="mw-redirect" title="XSD" href="https://en.wikipedia.org/wiki/XSD">XSD</a>&nbsp;data types compared to simpler serialization formats and offer modularity and reuse through&nbsp;<a title="XML namespace" href="https://en.wikipedia.org/wiki/XML_namespace">XML namespaces</a>.</p>
